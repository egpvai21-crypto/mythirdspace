01
Frontend — one PWA that survives into native

The web app is the primary surface for MVP and Beta; the Phase 3 native wrappers reuse the same bundle. Choose a framework that renders fast on mid-range Android, has great form + data primitives, and can be wrapped by Capacitor without rewrite.

FE-01 Framework

Next.js 15 (App Router) React 19 · TypeScript

Single framework for marketing pages, authenticated client portal, and coach universe. Server components keep dashboards snappy, client components handle booking interactivity.

Why
  • PWA-ready out of the box; Capacitor wraps the build for Phase 3 native.
  • Vercel deploy in under a minute per commit — matches "fast iteration."
  • Huge library ecosystem; any open question has a shadcn recipe.
Why not Remix / SvelteKit / Expo-web
  • Remix: great, but smaller UI-kit ecosystem for internal dashboards.
  • Expo-first would force mobile decisions we've deliberately deferred.

FE-02 UI system

Tailwind + shadcn/ui Radix primitives · Lucide icons

shadcn gives us owned, editable components instead of a black-box library. Lands the M3S design tokens (already in tokens.css) directly onto accessible primitives.

Why
  • Copy-paste components = easy to restyle in M3S earth palette.
  • Radix handles accessibility for modals, combobox, date picker correctly.
  • No lock-in; we own every file.
Why not Chakra / MUI
  • Heavy bundle, opinionated aesthetic fights the M3S brand.

FE-03 Data & forms

TanStack Query + React Hook Form + Zod

Query handles cache, retries, optimistic updates (critical for booking UX). RHF + Zod make the onboarding form, health vault uploads, and policy config maintainable and typed end-to-end.

Why
  • Zod schemas double as Supabase validators and AI tool-call schemas.
  • Optimistic booking feels instant; rolls back cleanly on policy rejection.

02
Backend — Supabase as the default, not the ceiling

Supabase is the unfair-advantage layer: one service gives us Postgres, auth, row-level security, realtime subscriptions, file storage, and edge functions. That removes the three things that normally slow a small team down — wiring auth, setting up realtime, and building a file-upload pipeline.

BE-01 Platform

Supabase Postgres · Auth · Storage · Realtime · Edge Fns

Managed Postgres with row-level security; auth with email + phone OTP; realtime channels we use for live calendar updates; storage buckets for health records and BCA photos.

Why
  • RLS gives client-side queries that are still safe — cuts huge volumes of API code.
  • Realtime Postgres changefeeds power the live calendar grid with no extra infra.
  • Phone + email OTP built in — ships M-01 in hours, not weeks.
Why not Firebase / custom Node+Postgres
  • Firebase: NoSQL makes relational coach↔client↔session logic painful.
  • Roll-your-own: six weeks of plumbing we don't need to buy yet.
PowersM-01M-04M-05M-06
BE-02 Database shape

Postgres + Drizzle ORM Supabase-managed

A real relational schema keeps coach ↔ client ↔ session ↔ note ↔ program joins honest. Drizzle gives us typed queries that match our Zod schemas, so AI tool calls and UI forms share one source of truth.

Why
  • Structured onboarding data is the precondition for AI summaries (B-01).
  • Migrations with Drizzle Kit are reviewable Git diffs, not dashboard clicks.
PowersM-02M-07B-01L-02
BE-03 Realtime calendar

Supabase Realtime Postgres changefeeds

Coach calendar + booking is built on a single sessions table; clients subscribe to the coach's channel, and every insert / update / delete reflects instantly in the grid. No separate socket server.

Why
  • Solves the "two clients book the same slot" race with DB-level constraints + live push.
  • Reuses the same auth + RLS; a client can only subscribe to coaches they're permitted to see.

BE-04 Background jobs

Inngest Event-driven · durable

Any work that shouldn't block a user request: AI summary generation, notification fan-out, BCA photo ingestion, nightly cal-sync, invoice roll-ups. Retries, concurrency caps, and cron built in.

Why
  • One event bus for M-12 notifications, B-01 AI, B-08 cal sync — not three systems.
  • Free tier covers the pilot; scales without re-architecting.
Why not BullMQ + Redis
  • Needs a Redis we'd have to run and monitor — violates "managed over self-hosted."

03
AI — structured inputs in, structured outputs out

The AI surface has two distinct shapes: profile summarisation (batch, runs on new intake data, coach reviews) and programming drafts (on-demand, inline, editable). Both need structured schemas — free-text onboarding will never summarise well.

AI-01 Primary model

Claude Sonnet 4.5 via Vercel AI SDK

Summarisation and programming-draft tasks are long-context and require careful clinical tone. Claude handles nuanced health language and refuses dangerous recommendations cleanly.

Why
  • Strong on structured JSON outputs when paired with Zod schemas.
  • Long context window holds full client history + session log for programming drafts.
  • Tool-calling lets the model query session notes it needs on demand.
PowersB-01L-02
AI-02 Secondary models

OpenAI for voice & vision Whisper · GPT-4o mini vision

Whisper transcribes voice coach-notes (hands full during a session). GPT-4o-mini vision is the fallback for BCA photo intake when Bluetooth isn't available.

Why
  • Pay-per-use, no model hosting; these are narrow, discrete tasks.
  • Multi-provider avoids single-vendor risk as models shift.
PowersM-07B-02
AI-03 Orchestration

Vercel AI SDK + typed tool calls

Unified client for Claude and OpenAI with streamed UI support. Tool calls use Zod schemas — the model can only ask for data that actually exists in our DB.

Why
  • Streamed profile summaries feel fast even when they take 8–10 s.
  • Type-safe tool schemas eliminate the "model hallucinated a field" class of bug.
PowersB-01L-02
AI-04 Evaluation

Braintrust or LangSmith

Every AI output is reviewable: log inputs, outputs, coach accept/edit/reject. Used as the eval loop for B-01 before we graduate to L-02.

Why
  • Coach edits become training signal — we know when a prompt stops working.
  • Required for health-adjacent AI: audit trail, not a black box.

Integrations — buy, don't build

Every integration on this list is a domain we should never build from scratch. Pick a vendor, pay for the time back, move on.

IN-01 Calendar sync

Nylas Google + Apple + Outlook

Two-way sync of coach personal calendars into the booking grid. One API, one billing line, all three major providers. Auto-blocks M3S slots when a coach has an external event.

IN-02 Wearables

Normalises HR, steps, sleep, workouts across every wearable into a single shape we can aggregate weekly. Avoids writing a different Bluetooth / OAuth flow per device.

06
Where this stack bites back

Every choice has a trade. These are the ones worth being eyes-open about.

Risk · lock-in
Supabase vendor risk

If Supabase's pricing or reliability shifts, we own a lot of RLS + auth logic tied to it. Mitigation: the data itself is plain Postgres; we can exit to a managed Postgres (Neon / RDS) and swap auth for Clerk in a week.

Risk · data
Health data is sensitive

Blood work, body composition, wearable data need encryption at rest (Supabase covers this) and strict RLS. Add a per-client share-audit log from day one — cheap to add now, painful to retrofit.

Risk · AI
AI quality without structured input

B-01 only works if M-02 onboarding is schema'd, not free text. Resist the temptation to ship a "free-text intake" shortcut in MVP — it creates a six-month hole in AI quality.

Risk · integrations
Nylas + Terra cost curves

Per-user monthly fees scale linearly with member count. Budget for the step at ~500 and ~2000 active connected users; consider direct OAuth integrations for one or two biggest providers past that.

Risk · iOS native
Web Bluetooth on iOS Safari

iOS Safari doesn't support Web Bluetooth. In MVP / Beta, BCA via Bluetooth only works on Android-web or the installed PWA; iOS users fall back to photo upload until the Capacitor wrapper ships in L-01.

Risk · velocity
Feature-flag discipline

PostHog flags make it easy to ship half-built features behind a toggle — easy enough that flag rot becomes its own tax. Retire flags weekly; don't let "temporary" flags become permanent config.
